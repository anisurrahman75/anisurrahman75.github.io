Migrating 2 TB from RDS to a Managed DBaaS in ~10 Hours

By Anisur Rahman · 2026

Moving a multi-terabyte production database between platforms is a race against two clocks: the migration window the business will tolerate, and the replication lag you have to close before cutover. A naive mysqldump piped into the target will technically work — and will also take days.

Parallelism is the whole game

The trick is to stop treating the database as one stream. Partition by table, and within large tables by primary-key range, then move each chunk on its own worker. A 2 TB dataset that crawls as a single pipe finishes in hours when it's fanned out across dozens of concurrent transfers saturating the network.

Snapshot for consistency, chunk for throughput, and let a control plane keep the workers honest.

A consistent starting point

Parallel readers only produce a correct copy if they all see the same point in time. A transactionally consistent snapshot gives every worker the same view, so chunks transferred in parallel still reassemble into a coherent database on the other side. After the snapshot lands, change-data-capture closes the gap until cutover.

// control plane hands chunks to a worker pool
for _, chunk := range plan.Chunks { // table + PK range
  pool.Submit(func(){ copyChunk(src, dst, chunk) })
}
pool.Wait() // ~2 TB in ≈10h

A control plane that can resume

Over ten hours, something will fail — a transient network blip, a throttled connection. A gRPC control plane tracks the state of every chunk, so a failed transfer retries just that range instead of restarting the whole migration. That resumability is what makes a multi-terabyte move safe to run against production.

The payoff: a 2 TB MySQL database moved from RDS to a Kubernetes-managed DBaaS in roughly ten hours, with a cutover measured in seconds.
