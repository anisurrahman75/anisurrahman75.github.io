Sharded Operators: Backing Up 1000+ Databases in Parallel

By Anisur Rahman · 2026

A single Kubernetes operator reconciling thousands of databases is a scaling problem hiding in plain sight. One controller, one work queue, one event loop — and suddenly it has to take a backup of 1000+ database instances inside the same window. The naive approach serializes, the window blows past its deadline, and backups start overlapping with the next cycle.

The bottleneck is coordination, not compute

The work itself is embarrassingly parallel — each database backs up independently. What doesn't scale is one controller tracking every job's state, retrying failures, and updating status. The fix is to shard the responsibility so no single reconciler owns everything.

Shard the operator, not just the workload — each shard owns a slice of the keyspace and runs its own bounded concurrency.

Bounded concurrency per shard

Within a shard, a worker pool caps how many backups run at once. Too few and the window is wasted; too many and you saturate the network and object storage. A token-bucket or semaphore in front of the backup plugins keeps the fan-out under control while still saturating available throughput.

// bounded fan-out across a shard's databases
sem := make(chan struct{}, maxConcurrent)
for _, db := range shard.Databases {
  sem <- struct{}{}
  go func(db DB) { defer func(){ <-sem }(); backup(db) }(db)
}

Failure isolation

Sharding also contains blast radius. A misbehaving database or a stuck upload only stalls its own shard's pool, not the whole fleet. Retries, validation, and failure recovery live at the per-job level, so one bad backup never blocks the other 999.

The result is a system where adding more databases means adding more shards — horizontal scaling instead of a single controller buckling under load.
