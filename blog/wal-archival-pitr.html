WAL Archival and Point-in-Time Recovery in the Cloud

By Anisur Rahman · 2026

Every durable database keeps a write-ahead log: before a change touches the data files, it is first appended to a sequential log. That ordering is what lets a database recover a consistent state after a crash. If you can capture and replay that log, you can do something more powerful — reconstruct the database as it existed at any moment in time.

Why PITR matters for managed databases

A nightly full backup answers "restore to last night." Point-in-time recovery answers "restore to 14:32:07, just before the bad migration ran." For a DBaaS serving production clusters, that difference is the gap between losing a day of data and losing a few seconds.

Continuous WAL shipping turns object storage into an always-growing recovery timeline.

Shipping the log continuously

The archival side is conceptually simple: as the database closes each WAL segment, an archiver uploads it to object storage. In Kubernetes this runs as a sidecar or a controller-managed process, streaming segments to any S3-compatible bucket. The hard parts are the edges — back-pressure when the upload is slower than the write rate, deduplication across failovers, and verifying that the archived timeline has no gaps.

# archive a closed WAL segment to object storage
wal-g wal-push pg_wal/000000010000000000000003
# later: restore the base backup, then replay to a target time
wal-g backup-fetch /var/lib/pgsql LATEST
recovery_target_time = '2026-06-30 14:32:07'

Restoring across a distributed cluster

Recovery is where the distributed-systems problems show up. The restore process has to fetch the right base backup, replay archived segments up to the requested timestamp, and then promote a new primary — all while the rest of the cluster agrees on who that primary is. A gRPC proxy that streams WAL between pods keeps the replay coordinated even when individual nodes come and go.

Done well, the whole thing is invisible: an operator declares a target time, and the platform handles fetching, replaying, and failover automatically.
